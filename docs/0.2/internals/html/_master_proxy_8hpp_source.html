53  /**
54  * \brief This class represents the master agent in a subagent program.
55  *
56  * \par Introduction
57  *
58  * This class is used on the subagent's side of a connection between
59  * subagent and master agent. It serves as a proxy which represents the
60  * master agent. It is possible for a subagent to hold connections to more
61  * than one master agents. For each connection one MasterProxy object is
62  * created. Multiple connections to the same master agent are possible,
63  * too, in which case one MasterProxy per connection is needed.
64  */
65  /**
66  * \par Connection State
67  *
68  * The MasterProxy is always in one of the following states:
69  * - connected
70  * - disconnected
71  * .
72  * The session to the master agent is established when creating a
73  * MasterProxy object, thus the object usually starts in connected state.
74  * If that fails, the object starts in disconnected state. A connected
75  * MasterProxy object may also loose the connection to the master agent
76  * and consequently become disconnected, without informing the user. It is
77  * possible to re-connect with the reconnect() function at any time (even
78  * if the session is currently established - it will be shut down and
79  * re-established in this case). When the object is destroyed, the session
80  * will be cleanly shut down. The connection state can be inspected with
81  * the is_connected() function. Some functions throw a disconnected
82  * exception if the session is not currently established.
83  *
84  */
85  /**
86  * \par The QApplication object
87  *
88  * The agentXcpp library uses the QT library. Processing SNMP requests
89  * (such as get and set) is performed within the QApplication event loop.
90  * Therefore, a program using agentXcpp must create a QApplication object
91  * and call exec() on it to start the event loop. The agentXcpp library
92  * also creates a QThread object for the networking part. Networking is
93  * done within the event loop of that thread.
94  *
95  */
96  /**
97  * \par Registrations
98  *
99  * Before the master agent sends requests to a subagent, the subagent must
100  * register a subtree. Doing so informs the master agent that the subagent
101  * wishes to handle requests for these OIDs. A subtree is an OID which
102  * denotes the root of a subtree in which the offered objects
103  * resides. For example, when two objects shall be offered with the OIDs
104  * 1.3.6.1.4.1.42<b>.1.1</b> and 1.3.6.1.4.1.42<b>.1.2</b>, then a subtree
105  * with OID 1.3.6.1.4.1.42<b>.1</b> should be registered, which includes
106  * both objects. The master agent will then forward all requests
107  * conecerning objects in this subtree to this subagent. Requests to
108  * non-existing objects (e.g. 1.3.6.1.4.1.42<b>.1.3</b>) are also
109  * forwarded, and the agentXcpp library will take care of them and return
110  * an appropriate error to the master agent.
111  *
112  * The function register_subtree() is used to register a subtree. It is
113  * typically called for the highest-level OID of the MIB which is
114  * implemented by the subagent. However, it is entirely possible to
115  * register multiple subtrees.
116  *
117  * Identical subtrees are subtrees with the exact same root OID. Each
118  * registration is done with a priority value. The higher the value, the
119  * lower the priority. When identical subtrees are registered (by the same
120  * subagent or by different subagents), the priority value is used to
121  * decide which subagent gets the requests. The master refuses identical
122  * registrations with the same priority values. Note however, that in case
123  * of overlapping subtrees which are \e not identical (e.g.
124  * 1.3.6.1.4.1.42<b>.1</b> and 1.3.6.1.4.1.42<b>.1.33.1</b>), the most
125  * specific subtree (i.e. the one with the longest OID) wins regardless of
126  * the priority values.
127  *
128  * It is also possible to unregister subtrees using the
129  * unregister_subtree() function. This informs the master agent that the
130  * MIB region is no longer available from this subagent.
131  *
132  * \internal
133  *
134  * The MasterProxy object generates a RegisterPDU object each time a
135  * registration is performed. These RegisterPDU objects are stored in the
136  * registrations member.
137  *
138  * When unregistering, the matching RegisterPDU is removed from the
139  * registration member.
140  *
141  * The registration member becomes invalid on connection loss. Since a
142  * connection loss is not signalled, the member cannot be cleared in such
143  * situations. Therefore, it is cleared in the connect() method if the
144  * object is currently disconnected. If connect() is called on a connected
145  * MasterProxy object, the registrations member is not cleared.
146  *
147  * \endinternal
148  */
149  /**
150  * \par Adding and Removing Variables
151  *
152  * Registering a subtree does not make any SNMP variables accessible yet.
153  * To provide SNMP variables, they must be added to the MasterProxy
154  * object, e.g. using add_variable(). The MasterProxy object will then
155  * dispatch incoming requests to the variables it knows about. If a request
156  * is received for an OID for which no variable has been added, an
157  * appropriate error is returned to the master agent.
158  *
159  * Added variables can be removed again using remove_variable(). This makes
160  * a variable inaccessible for the master agent.
161  *
162  * \internal
163  *
164  * The variables are stored in the member variables, which is a
165  * std::map<OidValue, shared_ptr<variable> >. The key is the OID for which the
166  * variable was added. This allows easy lookup for the request
167  * dispatcher.
168  *
169  * When removing a variable, it is removed from the variables member.
170  *
171  * The variables member becomes invalid on connection loss. Since a
172  * connection loss is not signalled, the member cannot be cleared in such
173  * situations. Therefore, it is cleared in the connect() method if the
174  * object is currently disconnected. If connect() is called on a connected
175  * MasterProxy object, the variables member is not cleared.
176  *
177  * \endinternal
178  *
179  */
180  /**
181  * \internal
182  *
183  * \par Internals
184  *

186  * UnixDomainConnector class. The MasterProxy implements the handle_pdu()
187  * slot, which is connected to the UnixDomainConnector::pduArrived()

189  *
190  * \endinternal
191  *
192  * \todo Describe timeout handling
193  * \todo Byte ordering is constant for a session. See rfc 2741, 7.1.1
194  */

00036     /**
00037      * \brief This class represents the master agent in a subagent program.
00038      *
00039      * This class is used on the subagent's side of a connection between 
00040      * subagent and master agent. It serves as a proxy which represents the 
00041      * master agent. It is possible for a subagent to hold connections to more 
00042      * than one master agents. For each connection one master_proxy object is 
00043      * created. Multiple connections to the same master agent are possible, 
00044      * too, in which case one master_proxy per connection is needed.
00045      *
00046      * The master_proxy is always in one of the following states:
00047      *
00048      * -# connected
00049      * -# disconnected
00050      *
00051      * The session to the master agent is established when creating a 
00052      * master_proxy object, thus the object usually starts in connected state.  
00053      * If that fails, the object starts in disconnected state. A connected 
00054      * master_proxy object may also loose the connection to the master agent 
00055      * and consequently become disconnected, without informing the user. It is 
00056      * possible to re-connect with the reconnect() function at any time (even 
00057      * if the session is currently established - it will be shut down and 
00058      * re-established in this case).  When the object is destroyed, the session 
00059      * will be cleanly shut down. The connection state can be inspected with 
00060      * the is_connected() function.  Some functions throw a disconnected 
00061      * exception if the session is not currently established.
00062      *
00063      * This class uses the boost::asio library for networking and therefore 
00064      * needs a boost::asio::io_service object. This object can either be 
00065      * provided by the user or created automatically. There are two 
00066      * constructors available therefore. The used object (whether auto-created 
00067      * or not) can be obtained using the get_io_service() function. If the 
00068      * io_service object was autocreated by a constructor, it will be 
00069      * destroyed by the destructor. If the user provided the io_service, it 
00070      * will NOT be destroyed by the destructor. It is possible to create 
00071      * multiple master_proxy objects using the same io_service object, or using 
00072      * different io_service objects.
00073      *
00074      * Receiving data from the master agent (requests or responses to requests) 
00075      * is done asynchronously and only works if io_service::run() or 
00076      * io_service::run_one() is invoked. However, some operations (such as 
00077      * registering stuff) invoke io_service::run_one() several times while 
00078      * waiting for a response from the master agent. If the io_service object 
00079      * is not used exclusively by the master_proxy object (which is entirely  
00080      * possible), this may complete asynchronous events before the library 
00081      * operation (e.g.  registering) is completed. Even the internal 
00082      * asynchronous reception calls io_service::run_one() while waiting for 
00083      * more data. If this behaviour is not desired, a separate io_service 
00084      * object should be used for other asynchronous I/O operations.
00085      *
00086      * \internal
00087      * The io_service_by_user variable is used to store whether the io_service 
00088      * object was generated automatically. It is set to true or false by the 
00089      * respective constructors and evaluated by the destructor.
00090      *
00091      * Receiving and processing PDU's coming from the master is done 
00092      * asyncronously. The receive() function is used as a callback for this 
00093      * purpose.  When connecting to the master agent (i.e. in the connect() 
00094      * function), an asyncronous read operation is started to read the PDU 
00095      * header. The receive() function is executed when a header arrived and 
00096      * reads the payload of the PDU synchronously, parses it (using 
00097      * PDU::parse_pdu()) and processes it.  Then it sets up an asyncronous read 
00098      * operation to wait for the next header. The disconnect() function cancels 
00099      * the current read operation before shutting down the session.
00100      */

00245             /**
00246              * \brief Register a subtree with the master agent
00247              *
00248              * Before the master agent sends SNMP requests to a subagent, the 
00249              * subagent must register some OIDs. Doing so informs the master 
00250              * agent that the subagent wishes to handle requests for these 
00251              * OIDs.
00252              *
00253              * This function registers a subtree (or MIB region). It is 
00254              * typically called for the highest-level OID of the MIB which is 
00255              * implemented by the subagent. The subagent will then receive 
00256              * requests for the registered OID and its subtree. For example, 
00257              * registering 1.3.6.1.4.1.42.13 registers all OIDs beginning with 
00258              * this one, e.g. 1.3.6.1.4.1.42.13.1.1.2 and 
00259              * 1.3.6.1.4.1.42.13.200.1.3.2.1.1.2.
00260              *
00261              * \param subtree The (root of the) subtree to register.
00262              *

00264              *                 Default is 127 according to RFC 2741, 6.2.3.  
00265              *                 "The agentx-Register-PDU".

00267              * \param timeout The timeout value for the registered subtree, in
